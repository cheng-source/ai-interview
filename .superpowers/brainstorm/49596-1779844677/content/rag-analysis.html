RAG 在面试系统中用在哪？

逐阶段分析，不为了用而用

逐阶段分析

阶段 LLM 本身能做到吗？ 需要 RAG？
简历解析 LLM 直接理解简历文本，提取结构化信息 ✗ 不需要
破冰 基于简历提取的信息生成个性化开场白 ✗ 不需要
技术面 — 出题 LLM 熟知编程概念、算法、系统设计，能按难度生成题目 ✗ 不需要
技术面 — 评估答案 LLM 自带充足技术知识，能判断代码正确性和设计合理性 ✗ 不需要
技术面 — 题库匹配 如果有大量自建题库，需要从题库中检索最匹配岗位的题目 ⚡ 可选
行为面 — 出题 LLM 能生成 STAR 方法论的行为题，结合 JD 定制 ✗ 不需要
候选人反问 候选人可能问公司福利、技术栈、团队结构——LLM 不知道这些 ✓ 有价值
综合报告 LLM 基于全量面试记录直接生成评分和评语 ✗ 不需要

结论

RAG 确实有价值的场景
  1. 候选人反问 —— 公司手册、福利制度、团队架构、技术栈文档，这些 LLM 不知道，必须 RAG
  2. JD→题库匹配 —— 如果公司有历史题库（几百题以上），用 embedding 检索比人工配题高效
不需要 RAG 的场景（占 80%）

出题、追问、评估、报告 —— LLM 的知识储备已经够用，加 RAG 反而增加幻觉风险和延迟。

⚠ 常见误区：把"题库"当成 RAG 场景。如果题目是结构化存储在 PostgreSQL 里的，直接 SQL 查询比向量检索更准确。

建议: MVP 阶段不加 RAG

  • 核心面试流程（出题、追问、评估）不需要 RAG
  • 候选人反问阶段初期可以限制为"暂不支持回答公司相关问题"
  • 后续按需在反问节点接入 RAG，LangGraph 的 Tool 机制天然支持
  • 题库匹配先用 SQL 搞定，量大了再考虑向量检索
