LangChain vs LangGraph — 到底差在哪？

以你的面试系统为例

核心区别：一条线 vs 一张图

只用 LangChain
Chain 线性执行
简历解析 → 开场白 → 技术问答 → 行为问答 → 生成报告
一条链走到底，难以中途改道
智能追问？得自己写状态机
const state = { stage, depth, score }
while (stage !== 'done') {
  const response = await chain.invoke({state, answer})
  state = parseNextState(response) // 自己管状态
}
⚠ 追问超过3层时状态管理开始失控
⚠ 没有内置恢复机制
⚠ 多分支逻辑散落在代码各处
LangChain + LangGraph
StateGraph 有向图
    [简历解析]
        ↓
    [开场破冰]
        ↓
    [技术轮] ──→ [追问] ──→ [下一题]
        ↓              ↑
    [行为轮] ──→ [追问] ──→ [下一题]
        ↓
    [反问环节]
        ↓
    [综合报告]
✓ 条件边：回答质量决定走"追问"还是"下一题"
✓ Checkpoint：每步自动存状态，断线恢复
✓ 图即文档：面试流程可视化、可测试

The Deciding Factor: 追问复杂度

场景 LangChain 够用？ LangGraph 必要？
固定题库、固定顺序
10道题按顺序来，每题给分		 ✓ 够用 不必要
1-2层简单追问
答对了跳下一题，答错了给提示再问一次		 勉强，需自己写状态管理 ✓ 推荐
深层动态追问 + 跨阶段跳跃
根据回答深度动态展开，技术轮发现短板后行为轮调整方向		 ✗ 会很痛苦 ✓ 核心场景
断线恢复、HR中途介入
候选人掉线10分钟后回来继续，HR在某节点审核		 ✗ 得自己实现 ✓ 内置支持

My Assessment

你的场景需要 LangGraph

  • 动态追问是你明确要求的能力 —— LangChain 的 Chain 模型是线性的，追问的分支逻辑必须自己造
  • 6阶段面试流程每个阶段内部都有循环（出题 → 回答 → 追问 → 判断是否深入）—— 这是典型的图结构
  • 断线恢复对 HR 系统是刚需 —— Checkpoint 机制开箱即用
  • 费用不高：@langchain/langgraph 的 JS SDK 免费开源，不增加任何成本

一句话：LangChain 处理"说什么"，LangGraph 处理"什么时候说"。你的面试系统两者都需要。
